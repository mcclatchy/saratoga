Related assets in Fuse templates

BY JAY PILGREEN

One of the major drawbacks to using the Fuse skin is that we don't have access to the Series Nav object. The skin is very simple on purpose, which allows us to implement it very quickly. This is its strength and also its weakness. In almost all instances, there are related assets or a story is a part of a package. Despite several different attempts, this has historically been painful. With the increased requests for landing pages, or even landing pages inside stories like we have been promoting to newsrooms, we needed something new to handle this issue.

The first requirement I set for myself was that a base concept could be implemented by our team as quickly as possible. I also knew that whatever I came up with needed to be extentable to allow for project-by-project alterations or enhancements. Of course, I also took into consideration all of our other standard pressure points like AMP, apps and workflow.

In order to make anything dynamic, meaning that updates to stories can be made automatically, I had to create a method to query our API. I created a program to do this that we can run on a timer. At whatever increment we set, the program will fetch new JSON data and save it on the server.

The second part is the display, and for this I've made a solution using a web component. My thinking on this approach is that by moving the landing page from a standalone story of its own to a part of a story, it's not critical that it works the same in all environments. Web components, due to their specification, have excellent backward compatibility. They are also easily created or modified, which fits my first requirement. I made a simple example below to show the possibilities.

Consider this version a default setup or the 15-minute version. The only thing needed from the market is the stories they want to include.

I've left a portion of the component open to manipulation, since it's inevitable that someone thinks that the word series isn't right for their project.

Web components won't work in AMP, and probably won't in the apps either. This is not so much of a problem because of the way the v1 version of the shadow DOM works. I can cherry pick pieces that I want to include if the web component works, and hide others. This allows us to create a nice backup that is totally editable in CUE.

In this sample, I have left 2 openings to alter the component: the base html and the html for each article. It gives us a good range of possibilities without having to duplicate all the AJAX code.
